## Iteration 1 - TASK-001: Create user profile page
- Implemented profile page with read-only and edit modes
- Added `updateProfile` to AuthContext (PATCH /api/v1/auth/me) to update user state on profile save
- Created Zod schema matching OpenAPI UserUpdate constraints (name: max 100, phone: max 20, avatarUrl: uri max 2048)
- Refactored into 3 components (ProfilePage, ProfileEditForm, ProfileReadView) to stay under complexity limit
- Updated getLinksForRole to support empty roles array (visible to all authenticated users)
- Added /profile route in AuthGuard and Profile link in navbar
- 18 component tests covering render, edit toggle, validation, API errors, field error mapping
- Learnings: ESLint complexity rule triggers at 25, extracting sub-components is the fix. AuthContextValue changes require updating all test files that mock it.

## Iteration 3 - TASK-003: Integrate provider analytics into dashboard
- Created `useProviderAnalytics` hook calling `GET /api/v1/provider/analytics` with providerAnalyticsKeys in keys.ts
- Extracted `AnalyticsSection` component into `components/provider/analytics-section.tsx` to keep dashboard under 500 lines
- Shows 3 analytics stat cards: total bookings, total revenue (formatted as USD), average occupancy (as %)
- Revenue-by-route breakdown with proportional horizontal bars (pure HTML/CSS, no chart library)
- Full loading skeleton, error state with retry, empty state for revenue-by-route
- 3 hook tests + 14 analytics section tests + 2 dashboard integration tests, all passing
- Learnings: Extracting analytics into separate component keeps both files under 200 lines and cleanly separates concerns. The analytics query is independent from the existing dashboard queries — error in analytics doesn't block the rest of the dashboard.
