## Iteration 1 - TASK-001: Create user profile page
- Implemented profile page with read-only and edit modes
- Added `updateProfile` to AuthContext (PATCH /api/v1/auth/me) to update user state on profile save
- Created Zod schema matching OpenAPI UserUpdate constraints (name: max 100, phone: max 20, avatarUrl: uri max 2048)
- Refactored into 3 components (ProfilePage, ProfileEditForm, ProfileReadView) to stay under complexity limit
- Updated getLinksForRole to support empty roles array (visible to all authenticated users)
- Added /profile route in AuthGuard and Profile link in navbar
- 18 component tests covering render, edit toggle, validation, API errors, field error mapping
- Learnings: ESLint complexity rule triggers at 25, extracting sub-components is the fix. AuthContextValue changes require updating all test files that mock it.

## Iteration 3 - TASK-003: Integrate provider analytics into dashboard
- Created `useProviderAnalytics` hook calling `GET /api/v1/provider/analytics` with providerAnalyticsKeys in keys.ts
- Extracted `AnalyticsSection` component into `components/provider/analytics-section.tsx` to keep dashboard under 500 lines
- Shows 3 analytics stat cards: total bookings, total revenue (formatted as USD), average occupancy (as %)
- Revenue-by-route breakdown with proportional horizontal bars (pure HTML/CSS, no chart library)
- Full loading skeleton, error state with retry, empty state for revenue-by-route
- 3 hook tests + 14 analytics section tests + 2 dashboard integration tests, all passing
- Learnings: Extracting analytics into separate component keeps both files under 200 lines and cleanly separates concerns. The analytics query is independent from the existing dashboard queries — error in analytics doesn't block the rest of the dashboard.

## Iteration 4 - TASK-004: Create driver passenger manifest
- Created `useDriverTripPassengers` hook calling `GET /api/v1/driver/trips/{scheduleId}/passengers` with date query param
- Added `passengers` key factory to `driverTripKeys` in keys.ts
- Created `components/driver/passenger-list.tsx` as separate component (trip-detail.tsx was already 613 lines)
- PassengerList shows name, boarding/alighting stops, seat labels, status badge per passenger
- Confirmed count / total seats in header, loading skeleton, error with retry, empty state
- Extracted `PassengerListContent` sub-component to stay under ESLint complexity limit of 15
- Replaced static passenger count card in trip-detail.tsx with PassengerList component
- Updated trip-detail.test.tsx to mock useDriverTripPassengers and adapt passenger info tests
- 4 hook tests + 17 component tests + 41 page tests (68 total), all passing
- Learnings: When adding a new hook to an already-mocked module, ALL existing tests that mock that module need the new export added to their mock factory or they will fail with "not a function" errors.

## Iteration 5 - TASK-005: Create admin hooks for users and audit logs
- Added 4 hooks to use-admin.ts: useAdminUsers (query), useUpdateUserStatus (mutation), useForceLogout (mutation), useAuditLogs (query)
- Updated adminKeys.auditLogs in keys.ts to include all filter params (userId, action, dateFrom, dateTo) matching the OpenAPI spec
- Used UserStatusAction type from generated types for type-safe action parameter
- Mutation hooks invalidate both list and detail/session keys on success, show descriptive toasts per action
- 17 new tests (25 total in use-admin.test.ts): all 4 query/mutation hooks with success, filters, error handling
- Learnings: Admin keys already existed in keys.ts but auditLogs only had page/pageSize — always check existing key factories match the full OpenAPI query params before using them.
