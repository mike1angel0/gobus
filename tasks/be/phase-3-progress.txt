## Iteration 1 - TASK-001: Create search service
- Implemented SearchService with searchTrips and getTripDetails methods
- Created domain entities (SearchResult, SeatAvailability, TripDetail, TripStopTime)
- Created Zod schemas matching OpenAPI spec (searchQuerySchema, searchResultSchema, seatAvailabilitySchema, tripDetailSchema)

- 16 unit tests covering: segment pricing, availability computation, stop ordering, case-insensitive search, pagination, seat booking status

- Serializers convert Date fields to ISO strings, tripDate uses .slice(0,10) for date-only format
- 15 integration tests covering: search with results, empty results, reversed stops filter, missing params, invalid date, pagination, trip detail with bookings, 404, no auth required, all seats unbooked

## Iteration 3 - TASK-003: Create booking service
- Implemented BookingService with create, listByUser, getById, cancel methods
- Exported pure validation functions (validateSeatsOnBus, validateStopOrder) for testability
- create uses prisma.$transaction with Serializable isolation to prevent double-booking
- toBookingWithDetails maps Prisma records to BookingWithDetails shape (extracts seatLabels from bookingSeats)
- 26 unit tests covering: seat validation (non-existent, disabled, blocked), stop order validation, create with price calc, seat conflict 409, schedule not found, list with pagination and status filter, getById ownership enforcement, cancel states
- Learnings: Mock $transaction by extracting txClient and having $transaction call the callback with it; reuse computeSegmentPrice from search.service for segment pricing; ownership returns 404 not 403 per spec

## Iteration 4 - TASK-004: Create booking API routes
- Implemented 4 authenticated endpoints: GET /api/v1/bookings (list), POST /api/v1/bookings (create), GET /api/v1/bookings/:id (detail), DELETE /api/v1/bookings/:id (cancel)
- Created Zod schemas (createBookingBodySchema, listBookingsQuerySchema, bookingSchema, bookingWithDetailsSchema) with .describe() on all fields
- Separate serializers for list (bookingSchema, no schedule) vs detail (bookingWithDetailsSchema, with schedule/route/bus)
- 19 integration tests covering: list with pagination, empty list, status filter, create 201, seat conflict 409, schedule not found, invalid body, invalid stop order, get detail, ownership 404, cancel, cancel already-cancelled, auth 401
- Learnings: Auth plugin calls prisma.user.findUnique to check user status — integration tests MUST include user mock in mockPrisma and set mockUserFindUnique.mockResolvedValue in beforeEach; PRD said cancel returns 204 but OpenAPI spec says 200 with BookingWithDetailsDataResponse — always follow spec (source of truth)

## Iteration 5 - TASK-005: Create tracking service
- Implemented TrackingService with updatePosition, getByBusId, getActiveByProvider methods
- Created BusTrackingData domain entity in domain/tracking/tracking.entity.ts
- updatePosition validates bus existence (404), driver assignment via schedule.findFirst (403), then upserts BusTracking record
- scheduleId inferred from driver's assigned schedule if not provided in input
- toTrackingData mapper extracts clean entity from Prisma record
- 11 unit tests covering: upsert success, scheduleId inference, explicit scheduleId, bus not found 404, driver not assigned 403, null tripDate, assignment query filters, getByBusId found/null, getActiveByProvider with results/empty
- Learnings: BusTracking has @unique on busId — use prisma.busTracking.upsert for create-or-update; tracking 403 is correct per spec (not 404 like ownership checks) because it's about role-based assignment, not resource enumeration
