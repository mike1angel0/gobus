## Iteration 3 - TASK-003: Configure rate limiting
- Registered @fastify/rate-limit globally (100 req/min per user, falls back to IP)
- Per-route overrides via config.rateLimit: auth routes (10/min per IP), search routes (30/min per IP)
- Added 429 handling in error-handler.ts since rate-limit plugin throws errors that reach the error handler
- RFC 9457 response with RATE_LIMITED code and Retry-After header
- Learnings: @fastify/rate-limit v10 throws errors that hit the error handler — must handle statusCode 429 explicitly. Don't use custom onRoute hooks; use config.rateLimit per-route instead. Use app.inject() with remoteAddress for consistent IP in tests.
- 4 integration tests: global headers, auth limit=10, search limit=30, 429 trigger
- All 350 unit tests pass, typecheck + lint clean

## Iteration 6 - TASK-006: Add audit logging hooks to existing services
- Created `src/api/plugins/audit.ts` Fastify plugin that decorates request with `request.audit()` helper
- The helper extracts IP from x-forwarded-for (first entry) or socket, and user agent from headers automatically
- Integrated audit calls into auth routes: REGISTER, LOGIN_SUCCESS, LOGIN_FAILURE, LOGIN_LOCKED, LOGOUT, PASSWORD_RESET_REQUEST, PASSWORD_RESET_COMPLETE, PASSWORD_CHANGE
- Login failure/locked events use try/catch around authService.login() to audit on error before re-throwing
- Integrated audit calls into booking routes: BOOKING_CREATED, BOOKING_CANCELLED
- Added try/catch wrapper in AuditService.log() to handle synchronous errors (e.g., when prisma.auditLog is undefined in test mocks)
- 13 integration tests covering: helper availability, IP extraction from x-forwarded-for, user agent capture, all 8 auth events, both booking events
- Learnings: `decorateRequest('audit', null)` needs `as unknown as FastifyRequest['audit']` type cast. AuditService.log() must wrap in try/catch (not just .catch()) because if prisma.auditLog is undefined, the synchronous property access throws before any promise is created. Booking create uses interactive $transaction (callback), not array-style.

- Added `DELETE /api/v1/admin/users/:id/sessions` endpoint in admin routes (requires ADMIN role, returns 204)
- Updated OpenAPI spec with `userSessions` path definition
- 6 unit tests for cleanup logic, 4 unit tests for revokeAllSessions, 4 integration tests for force-logout endpoint
- All 371 unit + 290 integration tests pass, typecheck + lint + spec:lint clean
- Learnings: Integration tests with auth plugin need `mockResolvedValueOnce` chaining for the auth lookup and then the service lookup on `user.findUnique`. The cleanup job uses `deleteMany` (not `updateMany`) for tokens since expired tokens have no value to retain.

## Iteration 8 - TASK-008: Admin user management endpoints
- Implemented 3 new admin endpoints: GET /api/v1/admin/users (paginated, filterable by role/status), PATCH /api/v1/admin/users/:id/status (suspend/unsuspend/unlock), GET /api/v1/admin/audit-logs (paginated, filterable by userId/action/dateRange)

- Added ACCOUNT_UNSUSPENDED and ACCOUNT_UNLOCKED audit actions
- All endpoints require ADMIN role (preHandler: [app.authenticate, requireAdmin])
- 16 new unit tests for listUsers, updateUserStatus, listAuditLogs; 17 new integration tests for all 3 endpoints
- All 387 unit + 308 integration tests pass, typecheck + lint + spec:lint clean
- Learnings: OpenAPI 3.1 uses `type: ['string', 'null']` instead of `nullable: true`. Zod `z.record()` requires 2 args in this version: `z.record(z.string(), z.unknown())`. The updateUserStatus integration test needs 3 chained mockResolvedValueOnce calls (auth lookup, updateUserStatus findUnique, revokeAllSessions findUnique) for the suspend path.
